TextCatch Landing — Session Log
================================
Date: 2026-06-23

OVERVIEW
--------
Took the TextCatch landing page (a single-file static smoke test) from a local
folder to a live, deployed site on Vercel with a working lead-capture email flow.

WHAT WAS DONE
-------------
1. Repository setup
   - Initialized a git repo in this folder (was not under version control).
   - Committed index.html and README.md.
   - Added .gitignore (.vercel, .DS_Store, .claude/) and excluded a stray
     local .claude/settings.local.json so it would not be published.
   - Pushed to GitHub: https://github.com/borys-ctrl/textcatch-landing (branch: main)

2. Initial Vercel deploy
   - Created Vercel project "textcatch-landing" (framework: Other, no build step).
   - Deployed to production. Live at: https://textcatch-landing.vercel.app

3. Lead-capture email flow
   - Added serverless function api/lead.js:
       * POST-only endpoint at /api/lead.
       * Reads Resend API key from env var RESEND_API_KEY (never hardcoded).
       * Emails borys@bestflooringhonolulu.com with subject "New lead at textcatch"
         and the submitted name, email, business, and website.
       * Sets reply_to to the lead's email; returns {"ok":true} on success.
       * Sends via Resend REST API using fetch (no npm dependency / no build step).
   - Updated submitForm() in index.html to POST the four form fields as JSON to
     /api/lead (fire-and-forget) before showing the thank-you view.
   - Set RESEND_API_KEY in Vercel (production environment).
   - Redeployed to production.
   - Verified: GET /api/lead -> 405; POST with test payload -> 200 {"ok":true}.

CURRENT STATE
-------------
- GitHub repo: https://github.com/borys-ctrl/textcatch-landing (main)
- Live site:   https://textcatch-landing.vercel.app
- Lead emails: working via Resend to borys@bestflooringhonolulu.com

NOTES / FOLLOW-UPS
------------------
- Sender is Resend's shared onboarding@resend.dev, which only delivers to the
  Resend account owner's address. To send from a custom domain (e.g. @textcatch.io)
  or to other recipients, verify a domain in Resend and update FROM in api/lead.js.
- The Resend API key was pasted into the chat session; consider rotating it in the
  Resend dashboard and re-running `vercel env add RESEND_API_KEY production`.
- RESEND_API_KEY is set for the production environment only. Add it to preview/
  development if the function should work on non-production deploys.
- GitHub repo and Vercel project are not linked for auto-deploy; redeploy manually
  with `vercel --prod` after pushing, or connect the repo in Vercel's Git settings.
- README TODO still open: point a custom domain (e.g. textcatch.io) at the deploy.

OVERVIEW
--------
Scoped TextCatch up from a smoke-test landing page to the real product. Built
Layer 1 (the branded chat widget) and drafted Layer 2 (the Twilio SMS backend).
Both are code-complete but UNDEPLOYED — gated on Twilio credentials.

WHAT WAS DONE
-------------
1. Layer 1 — Widget (built)
   - Forked widget-source.js (the "Kai" Shopify chat widget) into
     textcatch-widget.js: stripped the Shopify HTML wrapper, leaving a single
     self-contained vanilla-JS IIFE that injects its own CSS + DOM.
   - Rebranded via CONFIG: agentName "Tex", businessName "TextCatch",
     accent #16B57A (brand green from the landing page), text logo "TextCatch"
     (no avatar image yet), and TextCatch peek/ask/closing copy.
   - Genericized two BFH-specific bits: the document.title cleanup regex and the
     console.error label. Internal element IDs left as bfh-* by choice (invisible,
     no collision risk on textcatch.app — clean rename deferred).
   - Embedded on the landing page: <script src="/textcatch-widget.js" defer>
     before </body> in index.html.

2. Layer 2 — Backend + Twilio (drafted)
   - Added serverless function api/chat.js (POST-only at /api/chat):
       * Receives the widget's lead POST (firstName, phone, email, comment).
       * Sends two SMS via the Twilio REST API using fetch (no SDK / no build step),
         matching api/lead.js conventions (CommonJS, raw fetch).
       * SMS #1 -> visitor: confirmation copy (per CLAUDE.md).
       * SMS #2 -> OWNER_PHONE_NUMBER: lead-alert copy (per CLAUDE.md).
       * Reads TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, TWILIO_PHONE_NUMBER,
         OWNER_PHONE_NUMBER from env vars (never hardcoded).
       * Promise.allSettled so a bad visitor number can't block the owner alert;
         returns 502 only if BOTH texts fail; 500 "SMS not configured" if any env
         var is missing (safe even if accidentally deployed).
   - Wired the widget: CONFIG.webhookUrl "REPLACE_ME" -> "/api/chat" (same-origin).

3. Version control
   - This folder was a download of the repo (no .git). Re-initialized git, added
     the existing remote (github.com/borys-ctrl/textcatch-landing), and fetched —
     then laid this session's work on TOP of the existing history (reset --mixed
     origin/main) rather than a fresh root, so the June 23 commits were preserved.
   - Committed Layers 1 & 2 as 5691939 and pushed to origin/main.

CURRENT STATE
-------------
- Chain is wired end-to-end in code: widget -> /api/chat -> two Twilio texts.
- Committed + pushed to GitHub: origin/main @ 5691939 (June 23 history intact below).
- NOTHING DEPLOYED for this session's work. The widget + backend are inert until
  Twilio credentials exist and are set as Vercel env vars.
- Deliberately held the deploy so the live demo isn't a dead submit button.
